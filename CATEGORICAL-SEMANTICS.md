# Categorical Semantics of Monoidal Constructs

## Background

This framework is inspired by the wiring diagram formalism in Fong & Spivak's *An Invitation to
Applied Category Theory*. Chapter 2 introduces wiring diagrams as a general compositional framework
(operadic perspective — boxes with typed ports, nested hierarchically). Chapter 4 revisits them in
the context of resource theories, profunctors, and monoidal categories, where the emphasis shifts to
the monoidal structure of combining resources in parallel and sequencing transformations.

Our framework draws from both: the hierarchical nesting of boxes (chapter 2's operadic flavor) and
the resource-flow / monoidal-product perspective (chapter 4). The precise categorical home is
somewhere between the two, as described below.

## The Category

### Objects

Objects are finite lists of CloudFormation resource types:

- `[]` — the monoidal unit `I` (no wires)
- `[Bucket]` — a single wire
- `[Bucket, OAI]` — two parallel wires
- `[Distribution, Bucket, OAI, BucketPolicy]` — four wires

More precisely, objects are elements of the free commutative monoid generated by the set of
CloudFormation resource type strings. The monoidal product `⊗` is list concatenation (or multiset
union, since braiding gives us commutativity up to isomorphism).

### Morphisms

A morphism `f: [A₁, ..., Aₙ] → [B₁, ..., Bₘ]` is a box: something that consumes n typed wires
and produces m typed wires.

Examples:

- `mkBucket: I → [Bucket]` (generator)
- `encrypt: [Bucket] → [Bucket]` (transformer / endomorphism)
- `grantOaiRead: [Bucket, OAI] → [Bucket, OAI, BucketPolicy]` (wirer)
- `staticSite: [Distribution, Bucket, OAI, Certificate] → [Distribution, Bucket, Record]`
  (composite box)

### Monoidal Structure

The monoidal product `⊗` on objects is concatenation. On morphisms, `f ⊗ g` runs f and g in
parallel on disjoint wires.

Given `f: A → B` and `g: C → D`:

```
f ⊗ g: A ⊗ C → B ⊗ D
```

The unit object is `I = []` (empty wire bundle). The category is **strict** monoidal (the
associator and unitors are identities, since list concatenation is strictly associative with empty
list as unit).

### Symmetry (Braiding)

The braiding `σ_{A,B}: A ⊗ B → B ⊗ A` swaps wire bundles. In the implementation, braiding is
implicit — variable binding in the host language (TypeScript) handles wire reordering without
explicit swap boxes.

The braiding is symmetric: `σ_{B,A} ∘ σ_{A,B} = id_{A⊗B}`.

## Additional Structure

### Comonoid Structure (Copying and Discarding)

Every object A has:

- Comultiplication `Δ_A: A → A ⊗ A` (copy / fan-out)
- Counit `ε_A: A → I` (discard)

These satisfy the comonoid laws:

- `(ε_A ⊗ id_A) ∘ Δ_A = id_A` (discarding one copy recovers the original)
- `(id_A ⊗ ε_A) ∘ Δ_A = id_A`
- `(Δ_A ⊗ id_A) ∘ Δ_A = (id_A ⊗ Δ_A) ∘ Δ_A` (coassociativity)
- `σ_{A,A} ∘ Δ_A = Δ_A` (cocommutativity)

This makes the category **gs-monoidal** (garbage-and-sharing monoidal, in the terminology of
Gadducci & Heckel). The "garbage" is discarding; the "sharing" is copying.

### Partiality of Copying

Copying is NOT free in our model. When a wire is copied and both copies are independently modified
by downstream boxes, the results must be merged at synth time. This merge can fail (conflict on the
same property path).

Formally: the comonoid structure is **partial**. The composition `Δ_A ; (f ⊗ g)` is well-defined
only when the patches produced by f and g are compatible (non-conflicting). The category of valid
diagrams is a subcategory of the free gs-monoidal category — specifically, the subcategory where all
comonoid compositions satisfy the merge compatibility condition.

### The Patch Monoid

For each resource type T, the set of property patches forms a **partial commutative monoid** (M_T, ·, ∅):

- Elements are partial property assignments (subsets of the property tree)
- Identity ∅ is the empty patch (no properties set)
- Multiplication is deep merge: `a · b` is defined iff a and b don't conflict (same leaf path,
  different values), and when defined, produces the union of their assignments
- Commutativity: `a · b = b · a` (merge order doesn't matter)
- Associativity: `(a · b) · c = a · (b · c)` (when all are defined)

A transformer `f: [T] → [T]` corresponds to an element `p_f ∈ M_T` — the patch it applies.
Sequential composition of transformers corresponds to monoid multiplication:

```
patch(f ; g) = patch(f) · patch(g)
```

Two transformers **commute** iff their patches are compatible: `f ; g = g ; f` iff `p_f · p_g` is
defined.

### Confluence

The partial monoid is **confluent** in the following sense: if patches a, b, c are pairwise
compatible, then `(a · b) · c` and `a · (b · c)` are both defined and equal. This guarantees that
parallel merge (combining multiple independent modifications) is well-defined regardless of the
order in which patches are combined.

This is the algebraic reason the synth-time merge is deterministic: given a set of compatible
patches, any order of merging produces the same result.

## Generators and Relations (Presentation)

The category has a presentation by generators and relations.

### Generators (atomic morphisms)

1. **Resource generators:** `mkBucket: I → [Bucket]`, `mkFunction: I → [Function]`, etc.
2. **Transformers:** `encrypt: [Bucket] → [Bucket]`, `enableWebHosting: [Bucket] → [Bucket]`, etc.
3. **Wirers:** `grantOaiRead: [Bucket, OAI] → [Bucket, OAI, BucketPolicy]`, etc.
4. **Structural:** `σ_{A,B}` (braiding), `Δ_A` (copy), `ε_A` (discard) for all objects A, B.

### Relations (equalities between compositions)

1. **SMC coherence:** naturality of braiding, associativity and unitality of ⊗, etc.
2. **Comonoid laws:** as above.
3. **Idempotency** (where applicable): `encrypt ; encrypt = encrypt` (applying the same patch twice
   is the same as applying it once, since merge of identical patches is a no-op).
4. **Commutativity of non-conflicting transformers:** if transformers f and g apply to the same
   resource type and their patches don't conflict, then `f ; g = g ; f`. This is the key relation —
   it expresses that the merge semantics at synth time is equivalent to freedom of composition
   order for independent operations.
5. **Naturality of copy:** `Δ ; (f ⊗ f) = f ; Δ` (copying then transforming both = transforming
   then copying). This holds because both sides produce two copies with the same patch applied.

### The free category modulo relations

The valid programs in our framework are exactly the morphisms in the category presented by these
generators and relations. Two programs are equivalent (produce the same template) iff they are equal
as morphisms — i.e., one can be transformed into the other by applying the relations.

## String Diagrams

Every morphism in a symmetric monoidal category has a faithful **string diagram** representation:

- Wires = lines (strings)
- Boxes = labeled nodes with input ports (top/left) and output ports (bottom/right)
- `⊗` = vertical juxtaposition (parallel wires)
- `;` = horizontal sequencing (connecting output ports to input ports)
- `σ` = wire crossings
- `Δ` = wire splitting (one wire becomes two)
- `ε` = wire termination (a wire ends)

The key theorem: **two string diagrams represent the same morphism iff they are related by planar
isotopy** (continuous deformation without cutting wires or moving boxes past each other on the same
wire).

This has direct practical consequences:

1. The GUI can freely rearrange box positions without changing program semantics.
2. Two programs that produce isotopic string diagrams produce identical templates.
3. The visual representation IS the mathematical object (not merely a picture of it).

## Relationship to Fong & Spivak

### Chapter 2: Wiring diagrams as an operad

Chapter 2 presents wiring diagrams via the operad of wiring diagrams `W`. An operad captures the
nesting structure: a box can contain sub-boxes, wired together, forming a composite box that is
itself usable in a larger diagram.

This corresponds to our "boxes all the way down" principle. A composition of boxes IS a box. The
operadic structure gives:

- A set of box types (our morphism types)
- For each box type, a set of "fillings" (ways to decompose it into sub-boxes)
- Composition of fillings (nesting)

The operadic perspective is most useful for understanding the **hierarchical** structure: zooming in
and out in the GUI corresponds to moving between levels of operadic composition.

### Chapter 4: Resource theories and monoidal categories

Chapter 4 models resources via monoidal preorders and profunctors. A resource theory is a symmetric
monoidal preorder (V, ⊗, I) where objects are resource types and the ordering a ≤ b means "a can be
converted to b."

Our model is richer: we have actual morphisms (boxes), not just an ordering. But the spirit is the
same: resources combine in parallel (⊗) and can be transformed sequentially (composition). The
"convertibility" relation `a ≤ b` becomes "there exists a box `f: a → b`."

### Where we diverge

Fong & Spivak's resource theories are typically **preorders** (at most one morphism between any two
objects). Our category is not — there can be many different boxes from `[Bucket]` to `[Bucket]`
(encrypt, enableVersioning, enableWebHosting, etc.). This is because we care not just about
*whether* a conversion exists, but *how* it's done (which properties are set).

Their profunctor framework models open systems where internal behavior depends on the external
context. Our framework deliberately avoids this: boxes are context-free (pure functions of their
inputs). This keeps the semantics in the simpler world of ordinary monoidal categories rather than
profunctor-enriched ones.

## Possible Extensions

### Coproducts (conditional resources)

CloudFormation conditions create resources that may or may not exist. Categorically, this suggests
**coproduct** structure: `A + B` (resource is either of type A or absent). The `fnIf` mechanism
partially captures this at the property level, but true conditional resources would require
morphisms of type `A → B + I` (produce B or nothing).

This is a non-trivial extension. The category would need to become **distributive monoidal** (⊗
distributes over +). Whether this is worth pursuing depends on how central conditions are to real
usage.

### Fixed points (recursion)

If we wanted boxes that unfold recursively (e.g., generating a tree of resources), we'd need
**traced monoidal** structure (a feedback/loop operator). This is well-studied (Joyal, Street,
Verity — "Traced monoidal categories") and corresponds to the Int construction.

Not obviously needed for IaC, but could model things like recursive stack deployment or auto-scaling
groups that reference themselves.

### Functors to other categories (multi-target)

The morphisms in our category describe infrastructure composition abstractly. A **functor** from our
category to another (e.g., a category of Terraform configurations, or Pulumi programs) would give a
systematic translation between IaC frameworks. The categorical structure guarantees that any such
functor preserves composition — translated programs compose the same way.

### Enrichment (cost, latency, permissions)

If we wanted to track properties of morphisms (e.g., "this box requires IAM admin permissions" or
"this composition has latency X"), we'd enrich the category over an appropriate monoidal category of
annotations. This is the standard enriched category theory machinery.

## Summary

The framework inhabits a **symmetric gs-monoidal category with a partial comonoid structure**,
presented by generators (atomic boxes) and relations (commutativity of non-conflicting patches, SMC
coherence, comonoid laws). String diagrams provide a faithful visual language. The patch monoid
captures the merge semantics algebraically. The operadic perspective captures hierarchical nesting.
